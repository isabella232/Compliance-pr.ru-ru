# <a name="architecture-overview"></a>Обзор архитектуры

## <a name="what-is-microsoft-365"></a>Что такое Microsoft 365?

Microsoft 365 — это облачная, основанная на подписке версия Office, Windows 10, Enterprise Mobility + Security и Compliance. Microsoft 365 клиенты получают такие клиенты, как Outlook и Windows, и они также получают выгоду от служб, которые майкрософт хостов от их имени, таких как Exchange Online, Microsoft Teams и SharePoint Online. Все компоненты службы регулярно обновляются в рамках модели подписки, чтобы у наших клиентов был "вечнозеленый" продукт. Корпорация Майкрософт управляет инфраструктурой служб от имени клиентов, что означает, что корпорация Майкрософт несет ответственность за обеспечение безопасности инфраструктуры, которая хранит данные клиентов.

С точки зрения масштабирования в настоящее время мы используем около миллиона машин для Microsoft 365 служб. Инфраструктура, в которой используются эти службы, варьируется в различных аппаратных и виртуализированных средах Azure, Windows и Linux, а также на платформах с несколькими клиентами и выделенными платформами. Microsoft 365 является глобальным бизнесом, и наша инфраструктура распространяется в центрах обработки данных по всему миру, что позволяет нашим клиентам соответствовать требованиям оседлости данных и суверенитету.

Короче говоря, служба является сложной, работает в невероятном масштабе и требует тысячи инженеров Майкрософт для создания и обслуживания. Для нас приоритетной задачей является обеспечение безопасности всей этой инфраструктуры.

## <a name="how-does-microsoft-365-ensure-isolation-between-customer-tenants"></a>Каким образом Microsoft 365 изоляцию между клиентами-клиентами?

Облачные службы Майкрософт построены на предположении, что все клиенты потенциально враждебны всем другим клиентам. Чтобы должным образом изолировать клиентов друг от друга, Корпорация Майкрософт реализует различные технологии изоляции и элементы управления. Эти элементы управления предназначены для защиты от утечки информации или несанкционированного доступа к данным клиентов между клиентами и предотвращения негативного влияния действий одного клиента на службу для другого клиента.

Содержимое клиента логически изолировано в Microsoft 365 с помощью Azure Active Directory Azure AD. Проверка подлинности пользователей в Microsoft 365 проверяет не только удостоверение пользователя, но и идентификатор клиента, в который входит учетная запись пользователя, что не позволяет пользователям получать доступ к данным за пределами среды клиента. Чтобы дополнить логическую изоляцию Azure AD, содержимое клиента всегда шифруется в покое и в пути. Отдельные службы также могут предоставлять дополнительные уровни изоляции клиента, например SharePoint изоляцию данных клиента в отдельных зашифрованных базах данных.

## <a name="how-does-microsoft-365-engineer-resilient-services-that-avoid-single-points-of-failure"></a>Как Microsoft 365 инженера устойчивых служб, которые избегают единичек сбоя?

Корпорация Майкрософт разрабатывает и создает облачные службы, чтобы повысить надежность и свести к минимуму негативные последствия для клиентов перед лицом ошибок и проблем с нормальной операцией. Эта стратегия начинается с разработки сети, соединяющей наши географически распределенные центра обработки данных. Архитектура сети Майкрософт включает прямые взаимосвязи и несколько сетевых путей. Microsoft 365 службы используют эту избыточность для автоматического маршрутирования трафика из-за сбоев, чтобы повысить качество обслуживания.

На уровне службы Microsoft 365 стратегии устойчивости компании приоритеты устойчивости программного обеспечения. По возможности наши службы развертываются в активных и активных конфигурациях с автоматизированным мониторингом состояния службы, что позволяет службе обнаруживать и восстанавливать многие распространенные ошибки и сбои без вмешательства человека. Помимо активных и активных конфигураций, Microsoft 365 службы увеличивают устойчивость к ошибкам, обеспечивая развертывание службы в отдельных зонах сбоя, предотвращая, что неисправность в одной зоне не повлияет на доступность других зон.

Устойчивость к данным дополняет устойчивость служб, защищая целостность и доступность данных в Microsoft 365 службах. Наши службы используют локальное избыточное хранилище и геоизбытие для репликации копий данных клиентов в различные зоны сбоя. Если данные повреждены или потеряны в одной зоне сбоя, к ним можно получить доступ в другой зоне сбоя без потери доступности. Автоматическая проверка целостности автоматически восстанавливает данные, на которые влияет множество видов физического или логического повреждения. Microsoft 365 также предоставляет клиентам средства для восстановления данных, случайно удаленных или измененных клиентом в Exchange Online и SharePoint Online.

## <a name="how-does-microsoft-365-track-dependencies-and-prevent-unauthorized-external-system-connections"></a>Как Microsoft 365 отслеживание зависимостей и предотвращение несанкционированных подключений к внешним системам?

Microsoft 365 группы служб определяют критически важные компоненты системы и их зависимости в рамках управления непрерывности бизнеса. Кроме того, Microsoft 365 и отслеживает все внешние системные подключения, чтобы гарантировать, что в конфигурациях сетевого брандмауэра разрешены только авторизованные подключения. Microsoft 365, зависимостей и внешних подключений Microsoft 365 архитектуре информационной безопасности. Как архитектура информационной безопасности, так и соответствующие схемы потока данных пересматриваются и обновляются ежегодно как минимум, а также всякий раз, когда в систему внося значительные изменения.

Microsoft 365 проверяется регулярно и автоматически с помощью облачных инструментов для проверки согласования с нашими принципами безопасности и непрерывного тестирования функций изоляции и устойчивости. Архитектурная проверка работает, чтобы автоматически определять экземпляры, в которых текущее состояние службы отнесло от нужного состояния, заменая любые отклонения для проверки и смягчения. Цель проверки архитектуры заключается в обеспечении того, чтобы возможности безопасности нашей инфраструктуры служб продолжили функционировать так, как ожидалось.
