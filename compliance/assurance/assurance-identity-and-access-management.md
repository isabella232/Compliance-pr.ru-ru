# <a name="identity-and-access-management-overview"></a>Обзор управления удостоверениями и доступом

## <a name="how-do-microsoft-online-services-protect-production-systems-from-unauthorized-or-malicious-access"></a>Как сетевые службы Майкрософт защищают производственные системы от несанкционированного или вредоносного доступа?

Веб-службы Microsoft предназначены для того, чтобы позволить инженерам Корпорации Майкрософт управлять службами без доступа к контенту клиента. По умолчанию инженеры Майкрософт имеют нулевой постоянный доступ (ZSA) к контенту клиентов и не имеют привилегированного доступа к среде производства. Веб-службы Майкрософт используют модель Just-In-Time (JIT), just-Enough-Access (JEA) для предоставления инженерам группы служб временного привилегированного доступа к рабочим средам, когда такой доступ необходим для поддержки сетевых служб Майкрософт. Модель доступа JIT заменяет традиционный постоянный административный доступ процессом, позволяющим инженерам запрашивать временное повышение до привилегированных ролей, когда это необходимо.

Инженеры, назначенные группе служб для поддержки производственных служб, запрашивают право на учетную запись группы служб с помощью решения для управления удостоверениями и доступом. Запрос на получение права вызывает серию проверок персонала, чтобы убедиться, что инженер прошел все требования к облачному скринингу, завершил необходимые тренинги и получил соответствующее утверждение руководства до создания учетной записи. Создать учетную запись команды обслуживания для запрашиваемой среды можно только после выполнения всех требований соответствия. Чтобы сохранить право на учетную запись группы службы, персонал должен проходить обучение на основе ролей ежегодно и переэкранить каждые два года. Если эти проверки не будут завершены или не будут пройдены, права будут автоматически аннулированы.

Учетные записи группы служб не предоставляет никаких постоянных прав администратора или доступа к содержимому клиента. Если инженеру требуется дополнительный доступ для поддержки сетевых служб Майкрософт, он запрашивает временный повышенный доступ к ресурсам, которые требуются с помощью средства управления доступом под названием Lockbox. Lockbox ограничивает повышенный доступ минимальными привилегиями, ресурсами и временем, необходимыми для выполнения назначенной задачи. Если авторизованный рецензент одобряет запрос на доступ к JIT, инженеру предоставляется временная учетная запись только с привилегиями, необходимыми для выполнения назначенной работы. Эта временная учетная запись требует многофакторной проверки подлинности и автоматически удаляется по истечении утвержденного периода.

В момент запроса доступа к JIT jeA применяются права и роли Lockbox. Принимаются и передаются только запросы на доступ к активам в рамках прав инженера. Lockbox автоматически отклоняет запросы JIT, не входящие в сферу полномочий инженера и роли Lockbox, включая запросы, превышащие допустимые пороговые значения.  

## <a name="how-do-microsoft-online-services-use-role-based-access-control-rbac-with-lockbox-to-enforce-least-privilege"></a>Как интернет-службы Майкрософт используют управление доступом на основе ролей (RBAC) с Lockbox для обеспечения наименьших привилегий?

Учетные записи группы служб не предоставляет никаких постоянных прав администратора или доступа к содержимому клиента. Запросы JIT на ограниченные привилегии администратора управляются с помощью Lockbox. Lockbox использует RBAC для ограничения типов запросов на высоту JIT, которые могут выполнять инженеры, обеспечивая дополнительный уровень защиты для обеспечения наименьших привилегий. RBAC также помогает обеспечить разделение обязанностей, ограничив учетные записи группы служб соответствующими ролями.
Инженерам, поддерживающих службу, предоставляется членство группам безопасности в зависимости от их роли. Членство в группе безопасности не предоставляет привилегированный доступ. Вместо этого группы безопасности позволяют инженерам использовать Lockbox для запроса высоты JIT при необходимости для поддержки системы. Конкретные запросы JIT, которые может сделать инженер, ограничены членством в группах безопасности.

## <a name="how-do-microsoft-online-services-handle-remote-access-to-production-systems"></a>Как онлайн-службы Майкрософт обрабатывают удаленный доступ к производственным системам?

Компоненты системы сетевых служб Майкрософт находятся в центрах обработки данных, географически отделенных от оперативных групп. Сотрудники центра обработки данных не имеют логического доступа к системам онлайн-служб Майкрософт. В результате сотрудники службы Майкрософт управляют средой с помощью удаленного доступа. Персоналу группы служб, которым требуется удаленный доступ для поддержки сетевых служб Майкрософт, предоставляется удаленный доступ только после утверждения авторизованного руководителя. Все удаленные подключения используют протокол TLS, совместимый с FIPS 140-2, для безопасных удаленных подключений.

Сетевые службы Майкрософт используют безопасные рабочие станции администратора (SAW) для удаленного доступа группы служб, чтобы защитить среды онлайн-служб Майкрософт от компрометации. Эти рабочие станции предназначены для предотвращения преднамеренной или непреднамеренной потери производственных данных, включая блокировку USB-портов и ограничение программного обеспечения, доступного на рабочей станции Secure Admin, к необходимому для поддержки среды. Безопасные рабочие станции администратора тщательно отслеживаются и отслеживаются для обнаружения и предотвращения злонамеренного или непреднамеренного компрометации данных клиентов инженерами Майкрософт.

## <a name="how-does-customer-lockbox-add-additional-protection-for-customer-content"></a>Как клиент lockbox добавляет дополнительную защиту для контента клиента?

Клиенты могут добавить дополнительный уровень управления доступом к содержимому, включив клиентскую блокировку. Когда запрос на повышение блокировки включает доступ к содержимому клиента, клиент Lockbox требует утверждения от клиента в качестве последнего шага в рабочий процесс утверждения. Этот процесс предоставляет организациям возможность утверждать или отказывать в этих запросах, а также обеспечивает непосредственное управление доступом к клиенту. Если клиент отклоняет запрос на блокировку клиента, доступ к запрашиваемому содержимому будет отказано. Если клиент не отклоняет или не одобряет запрос в течение определенного периода, срок действия запроса автоматически истекает без получения microsoft доступа к содержимому клиента. Если клиент утвердит запрос, временный доступ Корпорации Майкрософт к содержимому клиента будет автоматически регистрироваться, проверяться и отменяться автоматически по истечении срока, назначенного для завершения операции устранения неполадок.
